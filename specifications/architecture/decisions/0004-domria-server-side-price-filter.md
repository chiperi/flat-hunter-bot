# ADR-0004: DOM.RIA — push price into the search; area/rooms client-side; opt-2 detail cache

**Status:** accepted
**Date:** 2026-07-09

## Context
DOM.RIA — 2-крокове API: `search` дає id-и, `info/{id}` — деталі (окремий запит на кожне оголошення).
Деталі — головний множник запитів, тож їх cap-имо (`DOMRIA_MAX_DETAILS`). Спершу ціна/площа/кімнати
фільтрувались **клієнтськи**, після фетчу деталей. Наслідок: при вузькому фільтрі (напр. «від 80 000
грн» — преміум, ~1% оголошень) обмежений бюджет деталей витрачався на випадкові найновіші, які потім
відкидались, і DOM.RIA майже ніколи не влучав (тоді як Rieltor фільтрує ціну в URL і влучає щоразу).

## Decision
**Ціну пушимо в сам запит DOM.RIA** (`price_from`/`price_to`) — перевірено, що API фільтрує по
**гривневому еквіваленту** (конвертує $/€), збігаючись із клієнтською логікою `priceArr["3"]`. Площа
й кімнати лишаються **клієнтськими** (публічне API не приймає ці параметри). **Opt-2 кеш**: деталі
тягнуться лише для **нових** id (яких ще нема в кеші), у межах `DOMRIA_MAX_DETAILS` за цикл; при
переповненні бюджету — warn-лог (spec review H-2/M-4).

## Consequences
- ➕ Обмежений бюджет деталей іде на **релевантні** (у ціновому діапазоні) оголошення — DOM.RIA
  доходить нарівні з Rieltor.
- ➕ Валютні преміум-оголошення ловляться коректно (конвертація на боці DOM.RIA).
- ➖ Площа/кімнати досі клієнтські: вузький **фільтр площі** може «з'їдати» бюджет — тому warn-лог +
  порада підняти `DOMRIA_MAX_DETAILS`. Ключ дедупу/кешу включає ціну (різні бюджети не діляться фетчем).
