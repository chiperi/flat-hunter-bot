# ADR-0002: One filter per user, queried across all sites

**Status:** accepted (supersedes the original "multiple profiles per user" brief)
**Date:** 2026-07-09

## Context
Початковий бриф передбачав кілька незалежних профілів на користувача (напр. «квартира собі» +
«гараж як інвестиція»), кожен зі своїм сайтом. На практиці поля фільтрів для наявних джерел
(DOM.RIA, Rieltor) ідентичні (operation/city/rooms/price/area), тож «профіль на сайт» означав би
дублювати ту саму конфігурацію, щоб покрити обидва сайти. Користувач хоче максимум покриття за один
раз, і йому байдуже, з якого сайту оголошення.

## Decision
**Один фільтр на користувача**, що опитується проти **кожного** ввімкненого джерела. `/newsearch`
робить upsert одного профілю (не додає новий); кожне сповіщення позначене джерелом (`sourceLabel`).
Legacy-дублікати (зі старої per-site моделі) згортаються в один при наступному збереженні. Дедуп
seen — неймспейсний per-site (`source:id`), тож одна квартира на двох сайтах трекається окремо.

## Consequences
- ➕ Простіший UX: критерії задаються раз, покриття — з усіх сайтів.
- ➕ `matchesCriteria` і планувальник спрощені (немає per-source профілів).
- ➖ Не можна задати **різні** критерії під різні сайти (напр. інший бюджет для DOM.RIA vs Rieltor) —
  свідомий компроміс; сьогодні поля однакові, тож користі від цього не було.
- ➖ Можливі дублікати **однієї** квартири, якщо вона є на кількох сайтах (різні id) — прийнятно;
  можна прибрати джерело через `SOURCES`, якщо шуму забагато.
